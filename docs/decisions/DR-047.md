# DR-047: Per-task model selection for fan-out lives at the SealBox broker

> **Born `proposed`** (acceptance is a separate human act — *MinSpec: Accept ADR*).
> Records the direction + the invariant shape; mandates SPEC-019 / cross-repo follow-ups.
> **No code** — SealBox is unbuilt, SPEC-019 in Specify/migration (DR-044 cascade).

## Context

Claude Code now ships **Workflow** + **background agents** — orchestrator-side fan-out
where each spawned agent carries its own `{model, effort}` (`agent(prompt, {model, effort})`,
plus the CLAUDE.md *Subagent Model & Effort Selection* table). This raised the question: with
the orchestrator already choosing a model per sub-task, does **Scrooge** — sitting at
`ANTHROPIC_BASE_URL` — still get to do the thing that *is* its value, pick the most
appropriate model/effort/thinking per task? And if not, do we need Scrooge or SealBox to
provide their own fan-out so Scrooge controls the routing end-to-end?

There are **two layers** at which "pick the right model" can happen, and they have different
owners:

1. **Orchestrator layer** — the model *writing* the `agent()` call / choosing Agent opts.
   Per-task picking **already happens here**, with full task context, so the orchestrator is
   the best-placed actor. Claude Code owns this. It is the bulk of the savings, and it is not
   Scrooge's.
2. **Proxy layer (Scrooge)** — on the wire, seeing the request *after* the model is chosen.
   To "pick" here, Scrooge must **override the caller's stated model** — which is the exact
   never-wrong label lie of [scrooge DR-016](../../../scroogellm/docs/decisions/DR-016-model-label-honesty-and-autoswitch.md):
   the workflow progress tree / agent metadata says `opus`, Scrooge served `haiku`, two labels
   contradict. The lie applies to CC subagents too, because a subagent's displayed model is a
   live label.

So CC fan-out moves the model-**picking** savings *up* to the orchestrator, out of Scrooge's
reach. The interesting move is not to claw it back — it is to notice **where a truthful
per-task pick IS available**: the headless SealBox broker.

## Decision

**1 — Under Claude Code, Scrooge does not pick the model.** That is the orchestrator's job
(already done). Scrooge's value under CC fan-out is the **label-safe** proxy layer only:
- **Cache** — fan-out is cache-gold: N finders share one system prompt + the same file
  context; a pipeline re-sends context per stage. Dedup → cache hits. No model swap, no lie.
- **Measure / attribute** — per-subagent / per-phase / per-workflow token+$ accounting, which
  CC's native cost display does not break down. The workflow is the ideal injection point for
  a per-agent `X-Scrooge-Tag` (scrooge DR-013 attribution dimensions; the caller-side tag is
  the known gap). This ledger is the moat.
- **Advise (post-hoc)** — "this workflow ran 8 opus agents on mechanical lookups; N suit
  haiku — fix in the script." Recommend, never silent-swap (DR-016 posture).

Scrooge MUST NOT override a caller-named model on a CC request. Same never-wrong wall as the
live session.

**2 — Do NOT build a fan-out orchestrator inside Scrooge.** Orchestration needs the full task
context Scrooge (a wire-level proxy) does not have; reimplementing it there is strictly worse,
still hits the label wall, and violates the moat principle (the moat is route/cache/measure,
**not** orchestrate).

**3 — The truthful per-task model-selection home is the SealBox broker.** SealBox is *itself*
the orchestrator — it plans and runs the fan-out — and its model call already funnels through
the **host-side broker** (DR-017). A headless autonomous run has **no live per-subagent picker
UI**, so choosing model/effort per sub-task *at the broker* does **not** lie. The broker is
therefore where Scrooge's per-task picking becomes both **truthful** and **end-to-end**:
SealBox fans out, the broker (Scrooge-aware) prices and routes each fanned-out call.

**4 — Reframe never-wrong: the constraint is label-truth, not a ban on model-picking.**
- *Live session* → recommend-and-one-keystroke, the human actuates `/model` (DR-016).
- *Headless broker* → truthful auto-pick is **permitted**, provided the model that actually
  ran per sub-task is **disclosed in the run's outcome record** (the headless analog of
  DR-016's footer + statusline honesty surface).

## Alternatives considered

- **Scrooge overrides the model on CC subagent calls (silent downgrade).** Rejected: the
  DR-016 lie — the workflow's per-agent label/metadata says one model, Scrooge served another.
- **Scrooge grows its own fan-out orchestrator.** Rejected: no conversation context,
  duplicates what CC already does better, still hits the label wall, breaks the moat principle.
- **Leave the broker model-agnostic (broker just relays whatever model SealBox names).**
  Rejected: forfeits the one surface where per-task picking is *both* truthful and end-to-end,
  i.e. the entire SealBox×Scrooge synergy. The broker is the chokepoint; make it the brain.

## Costly to Refactor

- **The broker↔Scrooge model-selection interface** — does the broker delegate the *pick*
  entirely to Scrooge, or does Scrooge return a recommendation the broker applies? Load-bearing;
  define with the broker, not after.
- **Outcome-record disclosure of the per-sub-task model** — the headless honesty contract
  (INV-16's counterpart). Must exist from v1; retrofitting "what actually ran" onto an
  autonomous substrate after the fact is the costly, never-wrong-regressing path (mirrors
  DR-016's footer-as-load-bearing-contract).

## Consequences

### Positive
- Clarifies Scrooge's role **per surface**: under CC → cache + measure + advise (not picking);
  under SealBox → route + pick (truthful). No more ambiguity about whether CC fan-out "cuts
  Scrooge out."
- Extends the never-wrong guarantee cleanly (DR-016 live-session → this DR headless-broker)
  without loosening it.
- Unblocks the SealBox broker design with an explicit model-selection responsibility.
- Even where Scrooge cannot pick (under CC), fan-out is cache-gold → real proxy-layer savings
  survive, and the advise layer still drives frugality.

### Negative / accepted
- Under CC, the per-task savings are the orchestrator's, not Scrooge's. Accepted: the
  alternative is a lying label, and cache+measure+advise is genuine value.
- SealBox's broker must become Scrooge-aware (wiring cost), and Scrooge needs a
  **headless/broker routing mode** distinct from the live-session recommend path
  (scrooge #33). Two cross-repo surfaces.
- **No code** — SealBox unbuilt, SPEC-019 in Specify/migration; Tier-0 untouched; no
  cred/egress added.
